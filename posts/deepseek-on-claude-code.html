Running DeepSeek
inside Claude Code

How I wired DeepSeek V4 Pro & Flash into the Claude Code CLI without losing my Anthropic Max subscription — and the limitation that forced the design.

claude → Anthropic Max claude-DS-pro claude-DS-flash → DeepSeek

iThe goal

Claude Code talks to Anthropic's API by default. But the CLI honors a handful of ANTHROPIC_* environment variables — point those at any Anthropic-compatible endpoint and the same CLI now drives a different model. DeepSeek ships exactly such an endpoint.

I wanted three commands on my machine:

claude

Untouched. Stays on my Anthropic Max subscription — no per-token billing.

claude-DS-pro

Same CLI, every request routed to DeepSeek V4 Pro.

claude-DS-flash

Same CLI, routed to the cheaper/faster DeepSeek V4 Flash.

1Get a DeepSeek API key

Create one at platform.deepseek.com/api_keys, top up a little credit, and store it as an env var. I keep mine in ~/.zshrc:

~/.zshrc
export DEEPSEEK_API_KEY=sk-your-key-here

Reload the shell after editing: source ~/.zshrc

2The env vars that do the work

DeepSeek exposes an Anthropic-compatible API. Setting these for a single command redirects that whole session:

VariableWhat it does
ANTHROPIC_BASE_URLPoints the CLI at api.deepseek.com/anthropic instead of Anthropic.
ANTHROPIC_AUTH_TOKENYour DeepSeek key (note: AUTH_TOKEN, not API_KEY).
ANTHROPIC_MODELDefault model for the session — deepseek-v4-pro or deepseek-v4-flash.
ANTHROPIC_DEFAULT_*_MODELWhat the CLI's Opus / Sonnet / Haiku tiers map to under the hood.
CLAUDE_CODE_SUBAGENT_MODELModel used for spawned subagents — kept on Flash for speed/cost.

3Wrap them in shell functions

Rather than export these globally (which would hijack every claude session), I scope them to two zsh functions. command claude ensures the real binary runs, and the vars exist only for that one invocation:

~/.zshrc
claude-DS-pro() {
  ANTHROPIC_BASE_URL=https://api.deepseek.com/anthropic \
  ANTHROPIC_AUTH_TOKEN="$DEEPSEEK_API_KEY" \
  ANTHROPIC_MODEL=deepseek-v4-pro \
  ANTHROPIC_DEFAULT_OPUS_MODEL=deepseek-v4-pro \
  ANTHROPIC_DEFAULT_SONNET_MODEL=deepseek-v4-pro \
  ANTHROPIC_DEFAULT_HAIKU_MODEL=deepseek-v4-flash \
  CLAUDE_CODE_SUBAGENT_MODEL=deepseek-v4-flash \
  CLAUDE_CODE_EFFORT_LEVEL=max \
  command claude "$@"
}

claude-DS-flash() {
  # same block, but every model = deepseek-v4-flash
  ANTHROPIC_BASE_URL=https://api.deepseek.com/anthropic \
  ANTHROPIC_AUTH_TOKEN="$DEEPSEEK_API_KEY" \
  ANTHROPIC_MODEL=deepseek-v4-flash \
  command claude "$@"
}

4Use it

terminal
source ~/.zshrc
claude             # Anthropic Max, as always
claude-DS-pro      # DeepSeek V4 Pro
claude-DS-flash    # DeepSeek V4 Flash
claude-DS-pro -p 'say hello in 3 words'   # one-shot smoke test

!Why not just add them to /model?

The obvious wish is to open one claude session and pick DeepSeek from the /model menu. I tried — here's why it doesn't work, and why the wrapper approach is actually the clean answer:

The picker is near-hardcoded. Claude Code's /model menu shows Anthropic's tiers (Opus / Sonnet / Haiku). Natively you can bolt on one extra entry via ANTHROPIC_CUSTOM_MODEL_OPTION — but only one. Two custom models (Pro and Flash) requires an external LLM gateway.
Max auth doesn't proxy. The Anthropic Max subscription authenticates through an interactive OAuth flow, not a static API key. It can't be passed through a local gateway. So any setup that routes everything through a proxy to mix providers loses Max access in that session — you'd be back to pay-per-token.
So the model lives in the launch command, not the menu. Each wrapper is its own clean lane: claude = Max, claude-DS-pro / claude-DS-flash = DeepSeek. No proxy daemon to babysit, nothing hijacked, fully reversible by deleting two functions. The "limitation" turned out to be the simpler design.
